## Message Shortcuts

We already [configured the message shortcut](https://pennow.tech/slack-app-
working-with-shortcuts/#Message_shortcut_configuration). Let's build an app
with the help of message shortcuts. Our app's purpose is to help us raise an
issue whenever any issue is reported. It'll have the following fields:

  * **Context information** : This will be retrieved automatically from the message context. The text could be like website down, failure, etc.
  * **#id** : A number assigned automatically. Displayed at the top.
  * **Assignee** : Whom to assign from the team
  * **Planned date** (optional): Till when to fix the issue
  * **Extra Details** (optional): Any detail which could be supportive.

Just like we did for _Global_ shortcuts, implement a shortcut handler for
message shortcuts in the same way. There won't be any change except `callback
id` which we configured for message shortcut:

    
    
    @app.shortcut("raise-website-issue-cb")
    def raise_issue_shortcut(ack, payload, client):
        ack()
    
        client.views_open(
            trigger_id=payload["trigger_id"],
            view= raise_ticket_view()
        )

## Implement view submission

Let's go a step further, and process the data submitted by the form. As we
discussed before that submitting a [slack modal view triggers `view()`
method](https://pennow.tech/slack-app-open-modal-
views/#Handling_job_submission) This method requires `callback_id` of the
view/form. Just _keep in mind_ that these view's callbacks are different from
callbacks we specified for shortcuts. view's `callback_id` can be retrieved
from the JSON which we copied from Slack Builder Kit.

    
    
    @app.view("ticket_create_view")
    def ticket_submitted(ack, payload, body, say):
        planned_date = payload['state']['values']['datepicker_block']['datepicker-action']['selected_date']
        if validate_date(planned_date) is False:
            ack({
                "response_action": "errors",
                "errors": {
                    "datepicker_block": "Planned date can't be in past"
                }
            })
        else:
            ack()
    
            say(process_submission_request(payload), channel='#web-monitor')
    

Just like before, we used `view()` method to handle view submission (line 1)
Then we verified that mentioned date isn't in past (line 4). If it's in past,
we'd throw an error(line 5) which will be displayed on the view form.
Otherwise, we move forward and parse the packet (line 14). and send the
response back. Following is the parsing logic to fetch the required values
from the payload:

    
    
    def process_submission_request(payload):
        values = payload['state']['values']
    
        # These details are retrieved from payload['blocks']
        url_val = payload['blocks'][0]['fields'][0]['text']
        url_status = payload['blocks'][0]['fields'][1]['text']
        ticket_id = payload['blocks'][2]['elements'][1]['text']
    
        # These details are retrieved from payload['state']['values]
        assignee_val = values['select_assignee_block']['select_assignee_action']['selected_option']['value']
        assignee_text = values['select_assignee_block']['select_assignee_action']['selected_option']['text']['text']
        planned_date = values['datepicker_block']['datepicker-action']['selected_date']
        extra_detail = values['extra_detail_block']['extra_detail_action']['value']
    
        return store_ticket(url_val, url_status, ticket_id, assignee_text, assignee_val, planned_date, extra_detail)
    
    

This was just a simple get and store function. Pay attention to the last
function. This is up to you to decide how you want to define this function.
You can either store it in some database or pass it to some message queues or
whatever you want. In our case, we just displayed these values back to the
user as a confirmation message.

## Practice at your end

This implementation is not foolproof. It has the scope of a lot of
improvements. Major improvement required here is to take care that the right
message is selected to raise a ticket. As the context menu can be invoked from
any shortcuts, so it might not have the required information which we need.
The result of selecting any such message will be that your program will not
work as expected. To implement logic to handle that. Another option is to
store values in a database The third option could be to increment the ticket
id every time a new ticket is raised. All of these are very simple to
implement assignments.
